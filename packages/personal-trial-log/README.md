# 개인 제작 기록 패키지

이 패키지는 하네스가 직접 사용하는 설계 자료가 아니다.

목적은 하네스 구축 과정에서 작성자가 겪은 시행착오뿐 아니라, 제작 프로세스의 전반적인 디테일을 개인 기록으로 남기는 것이다.

주의할 점은 이 패키지도 공개 저장소에 포함된다는 것이다. 따라서 "개인 기록"이라는 이름이 붙어 있어도 비공개 메모장이 아니다. 이곳에 남기는 모든 내용은 공개되어도 문제가 없는 수준으로 작성해야 한다.

여기서 제작 프로세스의 디테일은 다음을 포함한다.

- 어떤 순서로 생각을 정리하는지
- 어떤 지점에서 질문을 던지는지
- 어떤 기준으로 설계를 수정하는지
- 어떤 것을 공식 요구사항으로 남기고, 어떤 것을 개인 기록으로 분리하는지
- 구현 전에 어떤 불확실성을 제거하려 하는지
- 결과물보다 제작 루프를 어떻게 다듬고 있는지

## 기록 원칙

- 하네스 구현 로직이 이 문서를 참조하지 않게 한다.
- 공식 요구사항과 개인 회고를 섞지 않는다.
- 모든 기록은 공개 저장소에 올라갈 수 있다는 전제로 작성한다.
- 모든 기록은 상단 메타데이터에 `type`을 적는다.
- 시행착오는 실패 자체보다 "왜 그렇게 생각했고, 무엇을 바꿨는지"를 중심으로 기록한다.
- 제작 프로세스는 "어떤 순서와 기준으로 만들고 있는지"가 드러나게 기록한다.
- 나중에 같은 실수를 줄일 수 있을 정도로 구체적으로 쓴다.
- 감상문처럼 길게 쓰지 말고, 판단 변화가 드러나게 쓴다.

## 기록 타입

기록은 파일 위치보다 `type` 메타데이터로 구분한다.

| 타입 | 의미 | 판단 기준 |
| --- | --- | --- |
| `trial` | 시행착오 | 잘못된 전제, 놓친 요구사항, 실패한 판단, 바뀐 기준이 중심인 기록 |
| `process` | 제작 프로세스 | 작성자가 어떤 순서, 기준, 습관으로 만드는지가 중심인 기록 |
| `decision` | 의사결정 | 여러 선택지 중 하나를 고른 이유와 포기한 선택지가 중심인 기록 |
| `mixed` | 혼합 | 시행착오와 프로세스 관찰이 같은 비중으로 섞인 기록 |

구분 기준:

- "무엇을 잘못 봤고 어떻게 고쳤나"가 핵심이면 `trial`
- "어떤 방식으로 만들고 있나"가 핵심이면 `process`
- "왜 이 방향을 선택했나"가 핵심이면 `decision`
- 둘 이상이 분리하기 어려울 정도로 강하면 `mixed`

새 문서를 만들 때는 다음 메타데이터를 상단에 둔다.

```yaml
---
type: trial
created: YYYY-MM-DD
tags:
  - requirement
  - prototype
source: conversation
visibility: personal
---
```

## 기록 대상

다음 상황이 나오면 별도 기록으로 남긴다.

- 처음 설계한 원칙이 사용자 의도와 어긋난 경우
- 하네스가 요구사항을 임의로 가정하려 했던 경우
- 프로토타입 제작 경험 부족 때문에 놓친 개념이 드러난 경우
- 피드백을 받고 설계 방향이 바뀐 경우
- 구현 전에 물어봐야 할 질문을 뒤늦게 발견한 경우
- 자동화하면 안 되는 판단을 자동화하려 한 경우
- 하네스용 문서와 개인 학습 기록이 섞일 위험이 생긴 경우
- 사용자가 요구사항을 다듬는 방식이 반복적으로 드러난 경우
- 설계 문서를 먼저 만들고 구현으로 넘어가는 식의 제작 순서가 드러난 경우
- 질문, 기록, 설계, 구현 사이의 분리 기준이 생긴 경우
- 작업 중 새로 생긴 개인 운영 규칙이 있는 경우
- 하네스 구축자의 사고방식이나 의사결정 패턴이 보이는 경우

## 기록하지 않을 것

- 단순 작업 로그
- 파일 생성 내역
- 이미 공식 설계 문서에 반영된 요구사항의 중복 설명
- 감정 위주의 회고
- 하네스가 런타임에서 읽어야 하는 규칙
- API 키, 토큰, 비밀번호, 인증 정보
- 개인 이메일, 전화번호, 주소, 계정명처럼 공개할 필요가 없는 개인 식별 정보
- 회사, 조직, 스터디, 프로젝트의 비공개 내부 정보
- 비공개 회의 내용, 사적인 대화 원문, 제3자의 발언
- 로컬 경로, 서버 주소, 저장소 URL 중 공개하면 안 되는 정보
- 보안 설정, 권한 구조, 배포 환경처럼 공격에 활용될 수 있는 세부 정보
- 공개 저장소에 올리기 애매한 내용 전체

애매하면 기록하지 않는다. 필요하면 의미만 일반화해서 남긴다.

## 파일 구조

```text
packages/personal-trial-log/
  README.md
  records/
    index.md
  templates/
    trial-entry-template.md
    process-entry-template.md
```

## 기록 파일 이름

개별 기록이 필요하면 다음 형식을 쓴다.

```text
records/YYYY-MM-DD-type-short-title.md
```

예시:

```text
records/2026-07-08-trial-no-assumption-before-implementation.md
```

프로세스 기록은 다음처럼 쓴다.

```text
records/YYYY-MM-DD-process-short-title.md
```

예시:

```text
records/2026-07-08-process-separate-personal-records.md
```
