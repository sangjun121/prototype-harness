# 오픈소스 공개 전 보안 게이트 추가

## 관찰된 제작 프로세스

작성자는 하네스를 GitHub에 오픈소스로 공개할 계획을 전제로, 민감 파일과 민감정보가 저장소에 들어가지 않도록 검사하는 과정을 요구했다.

이 요구는 기능 구현 이후의 정리 작업이 아니라, 모든 변경 이후 실행되어야 하는 품질 게이트에 가깝다.

## 드러난 의사결정 기준

- 오픈소스 공개 가능성을 처음부터 전제로 둔다.
- 민감정보 검사는 사람의 주의에만 맡기지 않는다.
- 로컬 hook과 CI를 함께 사용해 우회 가능성을 줄인다.
- 외부 의존성이 없더라도 최소한의 기본 스캐너는 먼저 둔다.

## 공식 설계와의 관계

공식 설계에 반영해야 하는 결정이다.

하네스의 평가 시스템에는 민감정보 및 민감파일 검사가 포함되어야 한다. 프로토타입 생성 결과물도 공개 저장소에 들어갈 수 있으므로, 생성물과 문서 모두 보안 검사를 통과해야 한다.

## 반복 가능성

높다.

앞으로 파일 생성 기능, 템플릿 생성 기능, 실행 리포트 저장 기능이 추가될수록 보안 검사 범위를 계속 확인해야 한다.

## 다음 관찰 포인트

- 개인 기록 패키지에 공개하면 안 되는 개인 정보가 쌓이지 않는지
- 생성된 프로토타입 안에 API 키 예시가 실제 키처럼 들어가지 않는지
- 보안 검사가 너무 느리거나 noisy해서 개발 흐름을 방해하지 않는지
