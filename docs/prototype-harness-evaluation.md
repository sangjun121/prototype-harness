# 프로토타입 하네스 검증 체계

## 1. 목적

이 문서는 하네스가 만든 프로토타입이 좋은지 판단하는 기준을 정의한다.

하네스의 결과물은 단순히 "예쁜 화면"이나 "빌드되는 앱"으로 평가하면 안 된다. 이 하네스의 목표는 사용자의 요구사항과 핵심 아이디어를 바탕으로, 사용자가 떠올린 모습에 가까운 조작 가능한 프로토타입을 빠르게 만드는 것이다.

따라서 검증은 다음 질문에 답해야 한다.

```text
1. 사용자의 의도를 정확히 확인했는가?
2. 애매한 요구사항을 가정하지 않았는가?
3. 핵심 흐름이 실제로 조작 가능한가?
4. 사용자가 구체적인 피드백을 줄 수 있는 상태인가?
5. 다음 버전으로 개선할 수 있는 근거가 남았는가?
```

## 2. 평가 원칙

하네스 평가는 자동 지표와 사람 평가를 함께 사용한다.

자동 지표만으로는 의도 일치도를 알 수 없다. 반대로 사람 평가만으로는 매번 기준이 흔들린다.

따라서 다음 두 축을 모두 본다.

- 자동 검증: 빌드, 동작, 접근성, 반응형, 콘솔 에러, 핵심 플로우 실행 여부
- 사람 검증: 의도 일치도, 이해 가능성, 사용 흐름 자연스러움, 피드백 가능성

## 3. 핵심 평가 영역

### 3.1 요구사항 확인 품질

구현 전에 필요한 요구사항을 제대로 확인했는지 본다.

지표:

| 지표 | 설명 | 목표 |
| --- | --- | --- |
| 미확정 요구사항 차단율 | 미확정 항목이 있을 때 구현을 멈춘 비율 | 100% |
| 불필요 질문 비율 | 구현 판단에 영향이 적은 질문의 비율 | 20% 이하 |
| 핵심 질문 누락률 | 구현 후에야 필요성이 드러난 질문의 비율 | 10% 이하 |
| 위임 결정 명시율 | 사용자가 맡긴 결정이 명시 기록된 비율 | 100% |

판단 기준:

- 하네스가 임의로 요구사항을 채우면 실패다.
- 질문이 너무 많아도 실패다.
- 구현에 영향을 주는 질문을 놓치면 실패다.

### 3.2 의도 일치도

사용자가 원했던 방향과 결과물이 얼마나 가까운지 본다.

측정 방식:

프로토타입 사용 후 사용자에게 1~5점으로 묻는다.

```text
1. 내가 처음 떠올린 모습과 얼마나 가까운가?
2. 핵심 아이디어가 화면과 흐름에 반영되었는가?
3. 빼거나 바꿔야 할 방향 오류가 있는가?
```

목표:

- 평균 4.0점 이상
- 3점 이하 항목은 반드시 원인 기록
- 방향 오류 피드백이 나오면 다음 버전에서 최우선 수정

### 3.3 핵심 흐름 완결성

프로토타입이 주요 작업을 시작해서 끝낼 수 있는지 본다.

예시:

```text
기록 도구:
새 기록 작성 → 목록 반영 → 상세 확인 → 액션 아이템 체크
```

지표:

| 지표 | 설명 | 목표 |
| --- | --- | --- |
| 핵심 플로우 통과율 | Playwright 등으로 핵심 흐름이 끝까지 실행되는 비율 | 95% 이상 |
| 막힌 단계 수 | 사용자가 작업 중 멈춘 단계 수 | 1개 이하 |
| 죽은 CTA 수 | 눌러도 의미 있는 변화가 없는 핵심 버튼 수 | 0개 |
| 상태 변화 표시율 | 조작 결과가 화면에 반영되는 비율 | 100% |

### 3.4 피드백 가능성

사용자가 구체적인 피드백을 줄 수 있는지 본다.

좋은 결과물은 사용자가 이런 식으로 말하게 만든다.

```text
"기록 작성은 좋은데 질문 입력이 너무 깊숙이 있어요."
"목록에서 다음 액션 상태가 바로 보였으면 좋겠어요."
"공유 화면은 내가 생각한 것보다 보고서에 가까워요."
```

나쁜 결과물은 이런 반응을 만든다.

```text
"아직 잘 모르겠어요."
"좀 더 만들어봐야 알 것 같아요."
"예쁘긴 한데 뭘 봐야 할지 모르겠어요."
```

지표:

| 지표 | 설명 | 목표 |
| --- | --- | --- |
| 구체 피드백 비율 | 화면, 흐름, 정보, 조작 중 하나를 명확히 지적한 피드백 비율 | 70% 이상 |
| 모호 피드백 비율 | "괜찮다", "애매하다"처럼 실행으로 바꾸기 어려운 피드백 비율 | 20% 이하 |
| 다음 액션 도출률 | 피드백에서 다음 수정 항목이 나온 비율 | 80% 이상 |

### 3.5 시각 및 정보 구조 품질

디자인 자체보다 사용자가 화면을 이해하는 데 필요한 구조를 본다.

지표:

| 지표 | 설명 | 목표 |
| --- | --- | --- |
| 첫 화면 행동 인식률 | 사용자가 첫 화면에서 해야 할 일을 맞힌 비율 | 80% 이상 |
| 정보 과밀도 | 한 화면에 불필요한 정보가 많다는 피드백 비율 | 20% 이하 |
| 더미 데이터 현실감 | 더미 데이터가 실제 상황처럼 보인다는 평가 | 4.0점 이상 |
| 반응형 안정성 | 모바일/데스크톱에서 주요 요소가 깨지지 않는지 | 100% |

### 3.6 기술 품질

프로토타입이어도 기본 동작 품질은 확보해야 한다.

자동 검증:

- `npm run build`
- TypeScript 타입 체크
- ESLint
- Playwright 핵심 플로우 테스트
- 콘솔 에러 확인
- 모바일/데스크톱 스크린샷 확인
- 접근성 기본 검사

목표:

| 지표 | 목표 |
| --- | --- |
| 빌드 성공률 | 100% |
| 런타임 콘솔 에러 | 0개 |
| 핵심 플로우 테스트 | 통과 |
| 주요 뷰포트 레이아웃 깨짐 | 0개 |
| 접근성 심각 오류 | 0개 |

### 3.7 반복 개선 속도

하네스는 한 번에 완벽한 결과물을 만드는 도구가 아니다. 피드백을 받아 다음 버전으로 빠르게 개선하는 능력이 중요하다.

지표:

| 지표 | 설명 | 목표 |
| --- | --- | --- |
| 첫 프로토타입 생성 시간 | 확정 브리프 이후 실행 가능한 버전까지 걸린 시간 | 30분 이하 |
| 피드백 반영 시간 | 우선순위 피드백 1~3개를 반영하는 시간 | 20분 이하 |
| 재검증 통과율 | 수정 후 빌드/플로우 검증 통과율 | 95% 이상 |
| 회차별 의도 일치도 상승 | v1 대비 v2의 사용자 평가 상승 | +0.5점 이상 |

## 4. 종합 점수 모델

각 프로토타입 실행 결과는 100점 만점으로 평가한다.

| 영역 | 배점 |
| --- | --- |
| 요구사항 확인 품질 | 20 |
| 의도 일치도 | 25 |
| 핵심 흐름 완결성 | 20 |
| 피드백 가능성 | 15 |
| 시각 및 정보 구조 품질 | 10 |
| 기술 품질 | 10 |

등급:

| 점수 | 등급 | 의미 |
| --- | --- | --- |
| 90~100 | A | 바로 피드백 세션에 사용 가능 |
| 80~89 | B | 일부 보완 후 사용 가능 |
| 70~79 | C | 핵심 흐름은 있으나 의도나 사용성이 약함 |
| 60~69 | D | 피드백을 받기 전에 수정 필요 |
| 59 이하 | F | 하네스 결과물로 인정하지 않음 |

최소 통과 기준:

```text
총점 80점 이상
빌드 성공
핵심 플로우 통과
미확정 요구사항 0개
의도 일치도 4.0점 이상
```

## 5. 통계 관리

개별 결과물 점수보다 추세가 더 중요하다.

하네스는 실행마다 다음 데이터를 남겨야 한다.

```text
run_id:
date:
input_idea:
question_count:
confirmed_requirement_count:
unresolved_requirement_count:
build_passed:
core_flow_passed:
intent_match_score:
feedback_specificity_score:
overall_score:
time_to_first_prototype:
time_to_revision:
major_failure_reason:
```

주간 또는 10회 실행 단위로 다음 통계를 본다.

| 통계 | 의미 |
| --- | --- |
| 평균 종합 점수 | 전체 품질 추세 |
| 의도 일치도 평균 | 사용자 머릿속 모습에 가까워지는지 |
| 핵심 플로우 실패율 | 구현 안정성 |
| 질문 누락률 | 요구사항 확인 품질 |
| 모호 피드백 비율 | 피드백 가능한 결과물을 만들고 있는지 |
| 첫 프로토타입 생성 시간 중앙값 | 속도 |
| v1→v2 점수 상승폭 | 반복 개선 능력 |

주의:

- 평균만 보면 안 된다. 실패 사례를 따로 봐야 한다.
- 높은 점수보다 같은 실패가 반복되는지 보는 게 중요하다.
- 평가 대상이 5개 이하일 때는 통계보다 사례 분석이 더 중요하다.

## 6. 검증 도구

### 6.1 자동 검증 도구

초기 MVP에서 사용할 도구:

```text
npm run build
npm run lint
npm run typecheck
Playwright
axe-core
viewport screenshot
console error collector
```

역할:

| 도구 | 역할 |
| --- | --- |
| build | 앱이 실제로 빌드되는지 확인 |
| lint/typecheck | 기본 코드 품질 확인 |
| Playwright | 핵심 사용자 흐름 자동 실행 |
| axe-core | 접근성 심각 오류 확인 |
| screenshot | 레이아웃 깨짐과 빈 화면 확인 |
| console collector | 런타임 에러 확인 |

### 6.2 사람 검증 도구

초기에는 복잡한 리서치 도구보다 고정 질문지를 쓰는 편이 낫다.

필수 질문:

```text
1. 첫 화면에서 무엇을 하는 도구라고 이해했나요?
2. 내가 요청한 아이디어와 얼마나 가깝나요? 1~5점
3. 핵심 작업을 어디서 시작해야 하는지 바로 보였나요?
4. 직접 써보면서 막힌 지점이 있었나요?
5. 가장 먼저 고쳐야 할 한 가지는 무엇인가요?
6. 이 프로토타입을 보고 다음 의사결정을 할 수 있나요?
```

평가자는 최소 2명 이상이 좋다.

- 아이디어를 낸 사람 1명
- 실제 사용자에 가까운 사람 1명

## 7. 검증 시나리오

하네스는 고정된 벤치마크 입력으로 반복 검증한다.

초기 벤치마크:

```text
1. 스터디원이 매주 배운 내용과 질문을 남기고, 다음 주 액션 아이템을 정리하는 도구
2. 개발자가 사이드 프로젝트 아이디어를 빠르게 비교하고 우선순위를 정하는 보드
3. 카페 사장이 오늘 팔린 메뉴와 재고를 간단히 기록하는 모바일 앱
4. 면접 준비자가 예상 질문에 답변하고 피드백을 누적하는 훈련 도구
5. 여행 계획을 친구들과 함께 정리하고 후보지를 비교하는 도구
```

각 입력은 다음 절차로 검증한다.

```text
1. 하네스가 확인 질문을 만든다.
2. 사용자가 답변한다.
3. 확정 브리프가 생성된다.
4. 프로토타입이 생성된다.
5. 자동 검증을 통과한다.
6. 사람이 직접 사용한다.
7. 점수와 피드백을 기록한다.
8. v2를 만든 뒤 점수 상승폭을 확인한다.
```

## 8. 실패 분류

실패를 한 덩어리로 보면 개선이 어렵다. 반드시 원인을 분류한다.

| 실패 유형 | 의미 | 우선 조치 |
| --- | --- | --- |
| RequirementFailure | 필요한 질문을 안 했거나 답변을 잘못 반영함 | 입력 수집기 수정 |
| IntentFailure | 사용자가 원한 방향과 결과물이 다름 | 브리프 생성기 수정 |
| FlowFailure | 핵심 작업이 끝까지 이어지지 않음 | 화면 설계기 수정 |
| InteractionFailure | 눌러도 변화가 없거나 사용감이 어색함 | 구현 템플릿 수정 |
| DataFailure | 더미 데이터가 현실감을 주지 못함 | 데이터 생성 규칙 수정 |
| VisualFailure | 화면 구조가 이해를 방해함 | 디자인 시스템 수정 |
| TechnicalFailure | 빌드, 에러, 반응형 문제가 있음 | 검증기와 템플릿 수정 |
| FeedbackFailure | 피드백을 구체적으로 받기 어려움 | 피드백 질문 생성기 수정 |

## 9. MVP 목표

하네스 첫 버전은 프로토타입 생성 기능보다 평가 기능을 먼저 갖춰야 한다.

이유는 단순하다. 평가 기준이 구현되어 있지 않으면 하네스가 만든 결과물이 좋은지 나쁜지 매번 감으로 판단하게 된다. 그러면 하네스는 개선되는 시스템이 아니라 결과물을 계속 찍어내는 생성기가 된다.

따라서 MVP의 선행 조건은 다음이다.

```text
프로토타입 생성기를 만들기 전에:
- 평가 스키마가 있어야 한다.
- 자동 검증 러너가 있어야 한다.
- 사람 평가 설문 양식이 있어야 한다.
- 종합 점수 계산 방식이 있어야 한다.
- 실패 유형 분류 방식이 있어야 한다.
- 실행별 평가 기록 저장 방식이 있어야 한다.
```

하네스 첫 버전의 통과 목표는 다음이다.

```text
벤치마크 입력 3개 이상에서:
- 미확정 요구사항 0개 상태로 구현 시작
- 빌드 성공률 100%
- 핵심 플로우 통과율 90% 이상
- 의도 일치도 평균 4.0점 이상
- 종합 점수 평균 80점 이상
- v1에서 받은 피드백 중 1~3개를 v2에 반영
```

이 기준을 코드나 기록 시스템으로 측정할 수 없으면 하네스 MVP로 인정하지 않는다.

이 기준을 넘기 전에는 템플릿을 늘리지 않는다.

먼저 하나의 템플릿으로 평가 루프를 안정화하고, 그다음 템플릿 종류를 늘린다.

### 9.1 평가 시스템 구현 범위

MVP 전에 구현해야 하는 평가 시스템은 다음을 포함한다.

| 구성 요소 | 역할 | 필수 여부 |
| --- | --- | --- |
| evaluation-schema | 평가 결과 데이터 구조 정의 | 필수 |
| requirement-gate | 미확정 요구사항이 있으면 구현 차단 | 필수 |
| build-check | 빌드 성공 여부 기록 | 필수 |
| flow-check | 핵심 플로우 실행 결과 기록 | 필수 |
| human-review-form | 의도 일치도와 피드백 가능성 설문 | 필수 |
| score-calculator | 100점 만점 종합 점수 계산 | 필수 |
| failure-classifier | 실패 유형 분류 | 필수 |
| run-report | 실행별 평가 리포트 생성 | 필수 |
| trend-summary | 여러 실행 결과의 평균과 추세 요약 | 필수 |
| security-scan | 민감정보와 민감파일 포함 여부 검사 | 필수 |

초기에는 모든 검증을 완전 자동화하지 않아도 된다. 하지만 모든 평가 항목은 기록 가능해야 한다.

즉, 사람 평가가 필요한 항목은 수동 입력으로라도 남겨야 한다.

### 9.2 평가 시스템 없는 구현 금지

다음 상태에서는 프로토타입 생성 기능을 확장하지 않는다.

```text
- 종합 점수를 계산하지 못한다.
- 미확정 요구사항을 차단하지 못한다.
- 핵심 플로우 검증 결과를 기록하지 못한다.
- 사람 평가 결과를 저장하지 못한다.
- 실패 유형을 분류하지 못한다.
- 실행별 리포트를 남기지 못한다.
- 민감정보와 민감파일 검사를 통과하지 못한다.
```

이 원칙은 하네스의 품질 게이트다.

## 10. 결론

하네스의 결과물이 좋다는 것은 다음 네 가지를 증명하는 것이다.

```text
1. 하네스가 사용자 의도를 임의로 가정하지 않았다.
2. 사용자가 핵심 흐름을 실제로 조작할 수 있다.
3. 결과물을 보고 구체적인 피드백을 줄 수 있다.
4. 피드백을 반영한 다음 버전에서 점수가 올라간다.
```

이 네 가지가 측정되면 하네스는 감으로 개선하는 도구가 아니라, 반복할수록 나아지는 제작 시스템이 된다.
