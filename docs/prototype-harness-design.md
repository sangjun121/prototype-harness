# 프로토타입 제작 하네스 설계

## 1. 전제

이 하네스의 목적은 사용자가 요구사항이나 핵심 아이디어를 던졌을 때, 사용자가 떠올린 모습에 최대한 가까운 조작 가능한 프로토타입을 빠르게 만드는 것이다.

여기서 중요한 점은 프로토타입을 "완성도 낮은 제품"으로 보면 안 된다는 것이다. 프로토타입은 학습 장치다. 사용자의 머릿속 이미지, 실제 화면, 조작 흐름, 피드백 사이의 차이를 빨리 드러내는 도구다.

하네스는 프로토타이핑의 정신을 담은 결과물을 만들어야 한다. 여기서 프로토타이핑은 프로토타입을 사용자에게 부딪혀 반응을 보고, 무엇이 틀렸는지 배우고, 다음 결정을 내리는 학습 과정이다. 자세한 원칙은 `docs/prototyping-principles.md`에 둔다.

따라서 하네스는 코드를 빨리 생성하는 도구가 아니라, 다음 루프를 반복하는 시스템이어야 한다.

```text
아이디어 입력
→ 의도 복원
→ 불명확한 요구사항 질문
→ 사용자 답변으로 요구사항 확정
→ 빠른 구현
→ 직접 조작
→ 피드백 수집
→ 차이 분석
→ 수정
```

## 2. 프로토타입 제작에 필요한 핵심 지식

### 2.1 프로토타입의 목적

프로토타입의 가장 중요한 목적은 가장 중요한 가설을 세우고, 사용자에게서 유효한 피드백을 받는 것이다.

프로토타입은 보통 세 가지 중 하나를 검증한다.

1. 컨셉 검증
   - 이 아이디어가 사용자가 이해할 만한가
   - 첫 화면만 봐도 무엇을 하는지 알 수 있는가
   - 문제와 해결 방식이 맞물리는가

2. 흐름 검증
   - 사용자가 원하는 작업을 자연스럽게 끝낼 수 있는가
   - 다음 행동이 화면에서 명확한가
   - 중간에 헷갈리는 지점이 있는가

3. 인터랙션 검증
   - 버튼, 입력, 드래그, 선택, 필터, 편집 같은 조작이 기대와 맞는가
   - 반응 속도와 상태 변화가 자연스러운가
   - 사용자가 조작 결과를 즉시 이해할 수 있는가

하네스의 기본값은 `흐름 검증 + 인터랙션 검증`이어야 한다. 단순 정적 목업은 피드백 품질이 낮다.

다만 모든 경우에 웹 앱을 먼저 만들 필요는 없다. 핵심 경험을 더 싸게 검증할 수 있다면 종이 목업, 데모 영상, 역할극, 수동 운영, 기존 도구를 활용한 제한 실험이 더 적합할 수 있다.

### 2.2 좋은 프로토타입의 기준

좋은 프로토타입은 다음 조건을 만족한다.

- 사용자가 직접 눌러볼 수 있다.
- 핵심 사용 흐름이 첫 화면에서 시작된다.
- 더미 데이터가 실제 상황처럼 보인다.
- 사용자의 목표가 화면 구조에 반영되어 있다.
- 구현 범위가 작아도 "이게 어떤 제품인지" 느껴진다.
- 피드백을 받기 좋은 단위로 만들어져 있다.

나쁜 프로토타입은 다음과 같다.

- 설명 문구가 많고 실제 조작이 적다.
- 랜딩 페이지처럼 보이지만 사용 경험은 없다.
- 버튼은 있는데 상태 변화가 없다.
- 데이터가 너무 빈약해서 실제 사용 장면이 안 보인다.
- 사용자가 말하지 않은 기능을 과하게 붙인다.
- 시각적으로는 그럴듯하지만 검증할 질문이 없다.

### 2.3 프로토타입 범위 결정

프로토타입을 만들 때 가장 먼저 정해야 할 것은 기능 목록이 아니다. "이번에 무엇을 확인할 것인가"다.

필수 질문:

- 누가 쓰는가?
- 어떤 상황에서 쓰는가?
- 사용자는 무엇을 끝내고 싶은가?
- 가장 중요한 가설은 무엇인가?
- 이 가설을 가장 싸게 확인하는 방법은 무엇인가?
- 핵심 화면은 무엇인가?
- 이 프로토타입을 보고 어떤 피드백을 받고 싶은가?
- 실제 데이터가 필요한가, 가짜 데이터로 충분한가?
- 성공 판단 기준은 무엇인가?

이 질문에 답하지 않으면 하네스는 그냥 앱 생성기가 된다. 목표는 앱 생성기가 아니라, 의도에 가까운 프로토타입을 만드는 제작 루프다.

### 2.4 충실도 선택

프로토타입에는 충실도가 있다.

| 종류 | 적합한 상황 | 산출물 |
| --- | --- | --- |
| 저충실도 | 구조와 흐름만 확인 | 와이어프레임, 간단 UI |
| 중충실도 | 주요 조작과 화면 전환 확인 | 클릭 가능한 웹 앱 |
| 고충실도 | 실제 제품 느낌, 디자인 방향 확인 | 실제 데이터와 스타일이 있는 앱 |

이 하네스의 기본 목표는 중충실도다.

이유:

- 너무 낮으면 피드백이 추상적이다.
- 너무 높으면 제작 속도가 느리고 수정 비용이 커진다.
- 중충실도는 빠르게 만들면서도 조작 피드백을 받을 수 있다.

### 2.5 피드백 품질

피드백은 그냥 "어때요?"라고 물으면 품질이 낮다. 하네스는 피드백 질문을 함께 만들어야 한다.

좋은 피드백 질문:

- 첫 화면에서 무엇을 할 수 있다고 느꼈나요?
- 원하는 작업을 하려면 어디를 누를 것 같나요?
- 이 흐름에서 막힌 지점은 어디였나요?
- 기대한 결과와 실제 결과가 달랐던 부분은 무엇인가요?
- 이 화면에서 빠져도 되는 요소는 무엇인가요?
- 실제로 쓴다면 가장 먼저 고치고 싶은 부분은 무엇인가요?

나쁜 피드백 질문:

- 괜찮나요?
- 예쁜가요?
- 마음에 드나요?
- 더 필요한 기능이 있나요?

## 3. 하네스가 가져야 할 핵심 요소

### 3.1 입력 수집기

사용자의 원문 요구사항을 받는다. 이 단계에서 바로 구현하면 안 된다. 먼저 의도를 복원해야 한다.

수집해야 할 정보:

- 원문 아이디어
- 대상 사용자
- 사용 상황
- 핵심 작업
- 원하는 분위기나 레퍼런스
- 반드시 있어야 하는 기능
- 없어도 되는 기능
- 검증하고 싶은 질문

사용자가 적게 말했을 때 하네스가 빈칸을 임의로 채우면 안 된다. 구현에 영향을 주는 요구사항이 비어 있거나 애매하면, 사용자가 답할 때까지 질문해야 한다.

입력 수집기의 원칙:

- 구현에 앞선 요구사항은 절대 가정하지 않는다.
- 애매한 요구사항은 모두 사용자에게 다시 묻는다.
- 질문은 한 번에 너무 많이 던지지 않고, 구현 판단에 꼭 필요한 것부터 묻는다.
- 사용자가 답한 내용만 확정 요구사항으로 기록한다.
- 답변이 없는 항목은 `미확정`으로 남기고 구현 단계로 넘기지 않는다.
- 사용자가 명시적으로 "네가 정해줘"라고 위임한 경우에만 선택안을 제안할 수 있다.

예를 들어 "스터디 기록 앱을 만들고 싶다"라는 입력만으로는 다음을 바로 구현하면 안 된다.

```text
확인이 필요한 질문:
1. 기록은 개인용인가요, 팀 공유용인가요?
2. 기록 작성은 모바일 중심인가요, 데스크톱 중심인가요?
3. 반드시 들어가야 하는 기록 항목은 무엇인가요?
4. 이번 프로토타입에서 가장 먼저 확인하고 싶은 흐름은 무엇인가요?
```

### 3.2 의도 복원기

사용자의 말을 제품 요구사항으로 바꾼다.

출력 예시는 사용자 답변이 들어온 뒤에만 다음처럼 만든다.

```text
사용자는 "스터디 기록을 쉽게 남기고 공유하는 도구"를 원한다.
사용자 답변에 따르면 핵심 사용자는 팀 스터디 참여자다.
주요 목표는 매 회차마다 배운 내용, 질문, 다음 액션을 함께 남기는 것이다.
첫 프로토타입은 기록 작성 → 목록 확인 → 상세 확인 → 공유 흐름을 검증한다.
```

사용자 답변이 없으면 이 단계의 출력은 확정 브리프가 아니라 확인 질문이어야 한다.

의도 복원기는 다음을 구분해야 한다.

- 명시 요구사항: 사용자가 직접 말한 것
- 확인 필요 요구사항: 구현 전에 사용자 답변이 필요한 것
- 보류 요구사항: 아직 만들지 않는 것
- 위임된 결정: 사용자가 하네스에게 선택을 맡긴 것

여기서 `확인 필요 요구사항`이 남아 있으면 구현 단계로 넘어가면 안 된다.

### 3.3 프로토타입 브리프 생성기

구현 전에 짧은 제작 브리프를 만든다.

브리프 형식:

```text
목표:
대상 사용자:
핵심 사용 흐름:
첫 화면:
필수 기능:
더미 데이터:
디자인 방향:
검증 질문:
제외 범위:
확인 완료된 요구사항:
남은 미확정 항목:
```

이 브리프가 없으면 구현물이 산으로 간다. 하네스는 항상 브리프를 먼저 만들어야 한다. 단, `남은 미확정 항목`이 있으면 브리프는 초안 상태이며 구현을 시작하지 않는다.

### 3.4 화면 구조 설계기

최소한 다음 산출물을 만든다.

- 화면 목록
- 각 화면의 역할
- 주요 컴포넌트
- 상태 변화
- 화면 전환
- 빈 상태, 로딩 상태, 오류 상태

프로토타입이라도 상태가 있어야 한다. 상태가 없으면 사용자가 실제로 써보는 느낌을 받기 어렵다.

### 3.5 구현 생성기

기본 구현 방향:

- 웹 기반
- 로컬에서 바로 실행 가능
- 실제 API 없이 더미 데이터 우선
- 핵심 흐름은 끝까지 동작
- 디자인은 과장하지 않고 도메인에 맞게
- 첫 화면에서 바로 주요 작업 가능

권장 기술 스택:

- Vite
- React
- TypeScript
- Tailwind CSS 또는 CSS Modules
- lucide-react
- localStorage 기반 임시 상태

처음부터 백엔드, 인증, DB를 붙이면 속도가 느려진다. 프로토타입 하네스의 1차 목표는 실제 제품 구조가 아니라 사용 경험 검증이다.

### 3.6 검증기

하네스는 구현 후 스스로 확인해야 한다.

기본 검증:

- 앱이 빌드되는가
- 첫 화면이 비어 있지 않은가
- 핵심 버튼이 동작하는가
- 주요 흐름이 끝까지 이어지는가
- 모바일과 데스크톱에서 레이아웃이 깨지지 않는가
- 텍스트가 버튼이나 카드 밖으로 넘치지 않는가
- 더미 데이터가 실제 사용 장면을 보여주는가

가능하면 Playwright 같은 브라우저 검증을 붙인다.

### 3.7 피드백 패키지 생성기

프로토타입과 함께 피드백 질문을 자동으로 만든다.

출력 예시:

```text
이번 프로토타입에서 확인할 것:
1. 사용자가 첫 화면에서 핵심 행동을 바로 찾는가
2. 기록 작성 흐름이 충분히 짧은가
3. 목록과 상세 화면의 정보량이 적절한가

피드백 질문:
1. 첫 화면에서 무엇을 해야 할지 바로 보였나요?
2. 기록을 남기는 데 불필요하다고 느낀 입력이 있었나요?
3. 실제 스터디 후에 이걸 쓴다면 빠진 정보가 있나요?
```

## 4. 피드백 루프 설계

### 4.1 기본 루프

```text
1. 사용자 아이디어 입력
2. 하네스가 의도와 미확정 요구사항을 정리
3. 필요한 확인 질문 생성
4. 사용자 답변 수집
5. 프로토타입 브리프 확정
6. 구현
7. 자동 검증
8. 사용자 또는 피드백 대상자가 직접 사용
9. 피드백 수집
10. 수정 방향 분류
11. 다음 버전 생성
```

### 4.2 피드백 분류

피드백은 네 가지로 나눈다.

| 분류 | 예시 | 처리 |
| --- | --- | --- |
| 의도 불일치 | "내가 생각한 건 이게 아니야" | 브리프 수정 |
| 흐름 문제 | "어디를 눌러야 할지 모르겠어" | IA, 화면 전환 수정 |
| 정보 문제 | "필요한 정보가 안 보여" | 데이터 구조, 표시 방식 수정 |
| 사용감 문제 | "입력이 귀찮아" | 인터랙션, 기본값, 자동화 수정 |

이 분류가 중요하다. 모든 피드백을 기능 추가로 처리하면 프로토타입이 금방 무거워진다.

### 4.3 버전 관리

각 프로토타입 버전은 다음 기록을 남긴다.

```text
버전:
입력 아이디어:
확정 요구사항:
확인 질문과 답변:
검증 질문:
주요 변경:
받은 피드백:
다음 결정:
```

목적은 히스토리를 예쁘게 남기는 것이 아니라, 왜 이렇게 바뀌었는지 추적하는 것이다.

## 5. 하네스가 올바르게 작동하는지 검증하는 방법

상세한 검증 기준, 지표, 통계, 목표, 검증 도구는 `docs/prototype-harness-evaluation.md`에 둔다. 이 문서에서는 핵심 방향만 요약한다.

### 5.1 핵심 평가 기준

하네스의 성능은 코드 품질만으로 평가하면 안 된다. 다음 기준으로 봐야 한다.

1. 의도 일치도
   - 사용자가 "내가 생각한 것과 비슷하다"고 느끼는가

2. 제작 속도
   - 첫 조작 가능한 버전이 빠르게 나오는가

3. 피드백 가능성
   - 사용자가 구체적인 피드백을 줄 수 있는가

4. 수정 용이성
   - 피드백을 반영한 다음 버전이 빠르게 나오는가

5. 사용 흐름 완결성
   - 핵심 작업을 시작해서 끝낼 수 있는가

6. 불필요한 범위 통제
   - 처음부터 제품 전체를 만들려고 하지 않는가

### 5.2 테스트 과제

하네스를 만들면 아래 같은 입력으로 반복 테스트해야 한다.

```text
스터디원이 매주 배운 내용과 질문을 남기고, 다음 주 액션 아이템을 정리하는 도구
```

```text
개발자가 사이드 프로젝트 아이디어를 빠르게 비교하고 우선순위를 정하는 보드
```

```text
카페 사장이 오늘 팔린 메뉴와 재고를 간단히 기록하는 모바일 앱
```

```text
면접 준비자가 예상 질문에 답변하고 피드백을 누적하는 훈련 도구
```

각 입력에 대해 다음을 확인한다.

- 브리프가 명확한가
- 첫 화면이 핵심 흐름으로 바로 진입하는가
- 더미 데이터가 현실적인가
- 실제로 클릭하고 조작할 수 있는가
- 피드백 질문이 구체적인가
- 2차 수정이 쉬운 구조인가

### 5.3 실패 신호

하네스가 잘못 작동하는 신호:

- 매번 랜딩 페이지를 만든다.
- 요구사항보다 시각 효과에 집중한다.
- 사용자가 말하지 않은 기능을 많이 만든다.
- 핵심 흐름이 끝까지 동작하지 않는다.
- 더미 데이터가 빈약해서 사용 장면이 안 보인다.
- 검증 질문 없이 결과물만 만든다.
- 피드백을 받아도 어디를 고쳐야 하는지 분류하지 못한다.

## 6. MVP 설계

### 6.1 MVP 목표

첫 버전 하네스는 프로토타입 생성보다 평가 가능한 제작 루프를 먼저 갖춰야 한다.

평가 시스템이 없으면 하네스 결과물이 좋은지 증명할 수 없다. 따라서 프로토타입 생성 기능은 평가 게이트 위에서만 동작해야 한다.

첫 버전 하네스의 목표는 다음이다.

```text
사용자의 짧은 아이디어를 받아서,
필요한 확인 질문을 먼저 던지고,
답변을 바탕으로 명확한 브리프를 만들고,
클릭 가능한 중충실도 웹 프로토타입을 생성하고,
검증 질문과 다음 수정 방향을 함께 제공하고,
정해진 평가 기준으로 결과를 기록한다.
```

### 6.2 MVP 기능

필수 기능:

- 아이디어 입력
- 확인 질문 생성
- 사용자 답변 수집
- 확정 브리프 생성
- 화면 구조 생성
- React 프로토타입 생성
- 더미 데이터 생성
- 로컬 실행
- 빌드 검증
- 핵심 플로우 검증
- 평가 스키마
- 종합 점수 계산
- 실패 유형 분류
- 실행별 평가 리포트
- 사람 평가 입력 양식
- 민감정보 및 민감파일 검사
- Git hook 기반 보안 검사
- CI 기반 보안 검사
- 피드백 질문 생성
- 버전 기록

제외 기능:

- 실사용 백엔드
- 로그인
- 결제
- 복잡한 권한
- 배포 자동화
- Figma 연동
- 장기 데이터 저장
- 사용자 분석 SDK

### 6.3 내부 파이프라인

```text
InputIdea
→ ClarificationQuestions
→ ConfirmedRequirements
→ IntentBrief
→ PrototypeSpec
→ ScreenPlan
→ DataModel
→ ImplementationPlan
→ CodeGeneration
→ RuntimeVerification
→ FeedbackGuide
→ IterationRecord
```

각 단계의 책임:

| 단계 | 책임 |
| --- | --- |
| InputIdea | 사용자 원문 보존 |
| ClarificationQuestions | 구현 전에 필요한 질문 생성 |
| ConfirmedRequirements | 사용자 답변으로 확정된 요구사항 기록 |
| IntentBrief | 목표, 대상, 사용 상황 정리 |
| PrototypeSpec | 만들 것과 만들지 않을 것 결정 |
| ScreenPlan | 화면과 흐름 설계 |
| DataModel | 더미 데이터 구조 정의 |
| ImplementationPlan | 파일, 컴포넌트, 상태 설계 |
| CodeGeneration | 실제 프로토타입 구현 |
| RuntimeVerification | 빌드와 화면 검증 |
| FeedbackGuide | 피드백 질문 생성 |
| IterationRecord | 버전별 학습 기록 |

### 6.4 출력물 구조

초기에는 두 번째 구조보다 첫 번째 구조가 낫다. 매번 새 결과물을 만들고 비교하기 쉽기 때문이다.

## 7. 하네스 실행 흐름

### 7.1 사용자 입력

```text
스터디 기록을 쉽게 남기고 공유하는 도구를 만들고 싶어.
```

### 7.2 하네스 응답

```text
현재 입력만으로는 구현을 시작하지 않는다.
아래 항목이 확정되어야 프로토타입 브리프를 만들 수 있다.

확인 질문:
1. 이 도구는 개인 기록용인가요, 팀 공유용인가요?
2. 주 사용 환경은 모바일인가요, 데스크톱인가요?
3. 기록에 반드시 들어가야 하는 항목은 무엇인가요?
4. 공유는 실제 링크 공유까지 필요한가요, 아니면 공유용 요약 화면이면 충분한가요?
5. 이번 첫 프로토타입에서 가장 확인하고 싶은 흐름은 무엇인가요?
```

사용자가 답변한 뒤에만 다음처럼 확정한다.

```text
확정 요구사항:
- 팀 스터디원이 함께 쓰는 기록 도구다.
- 데스크톱 중심으로 만든다.
- 회차, 배운 내용, 질문, 다음 액션을 기록한다.
- 실제 링크 공유는 제외하고 공유용 요약 화면만 만든다.
- 첫 버전은 기록 작성 → 목록 확인 → 상세 확인 흐름을 검증한다.

이번 버전에서 만들 것:
- 기록 목록
- 새 기록 작성
- 기록 상세
- 액션 아이템 체크

이번 버전에서 제외할 것:
- 계정
- 알림
- 실제 공유 링크
- 파일 첨부
```

### 7.3 구현 후 제공

```text
실행 URL:
검증 결과:
피드백 질문:
다음 개선 후보:
```

## 8. 프로토타입 제작 원칙

하네스에 박아야 할 원칙:

1. 첫 화면은 설명보다 행동이어야 한다.
2. 핵심 흐름은 반드시 끝까지 동작해야 한다.
3. 더미 데이터는 현실적인 이름, 날짜, 상태, 수치를 가져야 한다.
4. 피드백을 받을 질문 없이 프로토타입을 끝내지 않는다.
5. 구현에 영향을 주는 요구사항은 절대 가정하지 않는다.
6. 애매한 요구사항은 구현 전에 사용자에게 다시 묻는다.
7. 사용자가 명시적으로 위임한 결정만 하네스가 대신 선택한다.
8. 기능 추가보다 의도 일치가 우선이다.
9. 고급 기술보다 빠른 조작 가능성이 우선이다.
10. 시각적 완성도는 도메인에 맞게만 올린다.
11. 매 버전마다 무엇을 배웠는지 남긴다.
12. 실패한 프로토타입도 학습 결과로 기록한다.

### 9.3 템플릿 전략

처음부터 모든 앱을 자유 생성하지 않는 편이 낫다. 도메인별 템플릿을 몇 개 두고, 그 위에 조정하는 방식이 빠르고 안정적이다.

초기 템플릿:

- 대시보드형
- 목록/상세형
- 보드형
- 폼 작성형
- 캘린더/스케줄형
- 훈련/퀴즈형

사용자 아이디어를 받으면 가장 가까운 템플릿을 고른 뒤, 화면 구조와 더미 데이터를 바꾼다.

### 9.4 검증 자동화

최소 검증:

```text
npm install
npm run build
npm run dev
브라우저 접속
첫 화면 스크린샷
핵심 버튼 클릭
모바일 폭 확인
```

브라우저 검증 체크:

- 첫 화면에 주요 CTA가 있는가
- 클릭 후 상태가 바뀌는가
- 목록, 상세, 작성 중 최소 2개 이상의 상태가 보이는가
- 콘솔 에러가 없는가
- 모바일에서 주요 버튼이 보이는가

## 10. 개발 순서

1. 설계 문서 확정
2. 프로토타입 브리프 포맷 확정
3. 첫 템플릿 하나 생성
4. 사용자의 아이디어를 브리프로 바꾸는 프롬프트/로직 작성

6. 빌드 검증
7. 피드백 질문 생성
8. 3개 테스트 과제로 품질 확인
9. 반복 기록 구조 추가
10. 템플릿 확장

## 11. 당장 만들 MVP

가장 현실적인 첫 구현은 다음이다.

```text
CLI 또는 간단한 웹 UI에서 아이디어 입력
→ brief.md 생성
→ React 템플릿 복사
→ src/data.ts, src/App.tsx, src/styles.css 생성
→ npm run build
→ feedback-guide.md 생성
```

처음부터 완전 자동 에이전트로 만들 필요는 없다. 먼저 "한 번의 입력으로 한 번의 프로토타입이 생기는 흐름"을 만든다. 그 다음 피드백 루프와 검증 자동화를 붙이는 편이 현실적이다.

## 12. 결론

이 하네스의 본질은 빠른 코드 생성이 아니다.

핵심은 다음 네 가지다.

```text
1. 사용자의 머릿속 의도를 구조화한다.
2. 검증할 질문을 먼저 정한다.
3. 조작 가능한 프로토타입을 빠르게 만든다.
4. 피드백을 다음 버전의 변경으로 연결한다.
```

이 네 가지가 없으면 하네스는 그냥 UI 생성기가 된다. 반대로 이 네 가지가 있으면, 처음 결과물이 완벽하지 않아도 매 반복마다 사용자가 생각한 모습에 가까워진다.
